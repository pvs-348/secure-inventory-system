About our Tool

This page demonstrates the complete secure record insertion lifecycle: record creation, message formatting, message integer conversion, RSA signing, signature verification, PBFT-style voting, consensus decision and final local storage.

Run the buttons from top to bottom one by one as all the steps depend on the previous steps.

Initial and Current Inventory State

Before adding a new record, this shows the current inventory state of Warehouses A, B, C and D. If the page is newly loaded and no record has been accepted yet, all nodes should show empty storage. If the record has been successfully run and accepted in the final PBFT consensus step, all nodes should show the new record in their storage.

Record Input

Choose the inventory node creating the new record. The selected node becomes the signer. The location is editable so incorrect sender-location attempts can be demonstrated.

Create Record

Creates the raw inventory record object from the form. No cryptographic protection has been applied yet.

Format Message

Converts the record into the consistent message format: itemId|itemQty|itemPrice|location. 

Waiting for formatted message.
3

Convert Message to Hash

Here MessageInteger was used instead of a standard hash function like SHA-256 or MD5, just to manually derive the functions and avoid importing external libraries. This converts the message string into a BigInt using base-256 conversion. We use the derived integer as the value signed by RSA. 

Waiting for message integer calculation.
4

Derive RSA Key Values

Uses the provided p, q, and e values of respective inventories to derive n, phi, gcd(e, phi) and private exponent d to further proceed signing and verification. 

Waiting for RSA derivation.
5

Sign Record

Signs the message integer using the originating node's private key. 

Waiting for signature.
6

Verify Signature

Verifies the signature using the originating node's public key. 

Waiting for verification.
7

Broadcast and Collect Votes

The originating node broadcasts the signed record package to all inventory nodes.
The package contains the record, origin node ID, formatted message, message integer and RSA signature.
Each node then independently checks:
(1) whether the signature is valid using the origin node's public key.
(2) whether the record fields are complete.
(3) whether quantity and price are positive.
(4) whether the record location matches the sender's identity.
Based on these checks, each node votes ACCEPT or REJECT.

Waiting for node votes.
8

Consensus and Storage

After all nodes vote, the system counts how many nodes voted ACCEPT and REJECT. In this simplified PBFT-style consensus, the record is accepted only if at least 3 out of 4 inventory nodes vote ACCEPT. This means one faulty or malicious node can disagree, but the system can still reach agreement. If the threshold is reached, the record is committed by storing a copy in every inventory node's local storage. If fewer than 3 nodes accept, the record is rejected and no storage is updated. 

Waiting for consensus result.
